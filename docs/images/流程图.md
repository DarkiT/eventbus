### 优雅关闭流程 (Shutdown)

```mermaid
flowchart TD
    A[调用 Shutdown] --> B[设置 shuttingDown 标志]
    B --> C[拒绝新的 Publish 请求]
    C --> D[等待进行中的发布完成]
    D --> E[遍历所有通道]
    
    E --> F[通知通道停止接收]
    F --> G[排空消息队列]
    G --> H[等待处理器自然完成]
    H --> I[取消上下文]
    I --> J[关闭消息通道]
    
    J --> K{通道全部处理完?}
    K -->|否| E
    K -->|是| L[清理全局资源]
    L --> M{超时?}
    M -->|是| N[强制关闭，返回 DeadlineExceeded]
    M -->|否| O[正常排空完成]
    
    N --> P[关闭完成]
    O --> P
    
    classDef start fill:#4CAF50,color:#fff
    classDef proc fill:#2196F3,color:#fff
    classDef drain fill:#FF9800,color:#fff
    classDef end fill:#9C27B0,color:#fff
    
    class A,P start
    class B,C,D,E,F,G,H,I,J,K,L proc
    class D drain
    class M,N,O end
```

> Shutdown 与 Close 的区别：Close 立即关闭不等处理中消息；Shutdown 等待已接收消息完成、排空队列后再关闭，适用于平滑发版/重启。

## TopicTrie 通配符匹配流程

### 基于 Trie 树的高效主题匹配

```mermaid
flowchart TD
    A[收到订阅/发布主题] --> B[标准化主题路径]
    B --> C[分割为层级 segments]
    C --> D{Trie 节点查找}
    
    D --> E[获取根节点]
    E --> F[遍历 segments]
    
    F --> G{当前 segment 匹配?}
    G -->|精确匹配子节点| H[进入子节点]
    G -->|匹配 + 通配符| I[记录通配符匹配节点]
    G -->|匹配 # 通配符| J[匹配该节点下所有处理器]
    G -->|无匹配| K[查找失败]
    
    H --> L{还有 segment?}
    L -->|是| F
    L -->|否| M[返回节点下的处理器列表]
    
    I --> L
    
    J --> M
    K --> N[返回空列表]
    
    M --> O[合并所有匹配处理器]
    N --> O
    O --> P[返回匹配结果]

    classDef proc fill:#2196F3,color:#fff
    classDef match fill:#FF9800,color:#fff
```

> TopicTrie 支持 MQTT 标准通配符：`+` 匹配单层级、`#` 匹配零或多层级。时间复杂度 O(n)，n = Topic 层级深度。
> 并发安全：读操作无锁（atomic.Value 快照），写操作用 COW 策略。
